World Turtle Dreams Post Mortem

Project Name: World Turtle Dreams Game
Project type: Processing Game Project
Time Frame: 25/6/2017 – 4/7/2017

This was a game I created as my first entry for #1GAM (One Game a Month). I started it very late in the month and so tried to keep it small and manageable for the time frame I had. I wanted to recreate and improve on one of my favourite flash games from my childhood – Jelly Blobs of Doom. I chose the Processing language for this project because I was already familiar with it from my college studies and it was a language I could prototype in quickly.

First, I boiled the game down to a few core components that I really wanted it to have:

  • -The screen should wrap around. If the player ran off the left side of the playing field they’d reappear on the right side as if they were on a giant moebius strip.
  • -The player character should be controlled by the w/a/s/d keyboard keys. The player should be able to play one handed, at no point should they require a mouse.
  • -The player should glide across the screen, their speed would decrease and increase slowly.
  • -The player would have to eat smaller circles (later clouds) to get bigger while avoiding larger circles that could swallow them up.

I also managed to include some extra features:

  • A lives system
  • Powerups
  • Score
  • Pixel art graphics and some sound effects

Initially I wanted to release the game on Windows as a standalone .exe file. After considering different options I chose to make a web version as well because this would make my game more accessible.

The Good:
  • The actual plan for the game was very manageable even with my limited experience and time.
  • I managed to get all the features I wanted into the game and even improve on the game I based it on, creating powerups to add some extra entertainment to the rather slow early game.
  • The final collision script works very well.
  • I like the way the pixel art turned out, especially since it was my first time making pixel art.

The Bad:
  • I had some difficulty creating a web version of the game. I used Processing.js to create a javascript version of the processing script. I had never used Processing.js and a lot of the quirks of porting the game gave me some trouble. Read a more in-depth analysis of the exact problems and how I fixed/worked around them here.
  • My initial attempts at writing a collision script were less than successful. I tried defining the player character in terms of an arrayList of points and then checked each frame if any point of the player was within in the boundary of any circle on the screen each frame. This was incredibly inefficient as the collision only registered about a quarter of the time.

The learning experience:
  • Leaving porting of a game to the last minute, instead of building it for the system ground up (or at the very least being aware of the system’s limitations) is not a good idea, especially when working with new tools. In the future, I should give myself more time in advance to get to know the systems I’m working with.
  • I can afford to be more ambitious in my plans for games.

Overall, I’m happy with the final product of the game I made and look forward to making another next month. Links to the game and code below.
